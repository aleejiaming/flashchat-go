🏆 學習里程碑 01：架構重構與 WebSocket 工廠模式

📅 紀錄時間：(請填寫今日日期)
🎯 目前進度：成功將原本全部擠在 main.go 裡的 WebSocket 聊天室，重構為「乾淨架構 (Clean Architecture)」，並順利導入了工廠模式處理 /help 與 /weather 機器人指令。

🧗 本次遭遇的三大困難與痛點覆盤

在這次重構中，我遇到了幾個巨大的觀念卡關，以下是這些痛點的紀錄與白話文解答，留給未來的自己複習：

痛點一：工廠模式的不熟悉 (結構體與參數傳遞)

我的盲點：不清楚 struct (結構體) 之間怎麼互相呼叫，以及介面 (Interface) 的參數到底怎麼傳。一直想把所有的變數全域複製貼上。

突破觀念 (依賴注入)：

在 Go 語言中，我們不要用「全域變數」。

「依賴注入 (DI)」的白話文：就是「把工具塞進員工的口袋裡」。例如服務生 (WSHandler) 需要廣播，我們就在他的 struct 裡開一個 Hub *ws.Hub 的口袋。然後在 main.go 聘請他時，把經理 (hub) 當作參數傳給他 (NewWSHandler(hub))。

工廠模式的本質：就是一個負責「看條件，然後給出不同工具」的 GetProcessor() 函式。這讓我們的 Hub 廣播中心完全不需要知道天氣機器人是怎麼寫的，只要會呼叫工廠就好。

痛點二：專案架構分類 (Handler, WS, Main 到底放什麼？)

我的盲點：拆分檔案時一竅不通，不知道哪段程式碼該歸誰管，導致互相 import 報錯。

突破觀念 (關注點分離)：
未來的專案一律照著這個「餐廳法則」來拆：

main.go (總部組裝線)：只能有 New... 初始化和啟動伺服器的代碼，絕對不能有 if/else 商業邏輯。

handler/ (外場服務生)：只負責「面對網路 (HTTP/WebSocket)」。把客人傳來的 JSON 拆開，丟給後台；或是把後台做好的資料轉成 JSON 還給客人。他不思考、不做菜。

ws/ (內場狀態與主廚)：hub.go 負責管理點名冊和通道 (狀態層)；processor.go 負責檢查字串是不是 /weather (邏輯層/Service)。

痛點三：HTTP 後端連接網路的實際方法與思路

我的盲點：對後端是怎麼跟網路接軌的，只有模糊的概念，缺乏實際手感。

突破觀念 (協定升級)：

傳統 HTTP：客人點一次餐 (GET /menu)，伺服器給一次菜單，然後馬上掛電話斷線。這叫無狀態 (Stateless)。

WebSocket：客人在 index.html 的 JavaScript 寫了 new WebSocket(...)。當這個請求打到後端的 handler 時，我們用了 upgrader.Upgrade(w, r, nil) 這行魔法。

升級的意義：這行魔法的作用是告訴瀏覽器：「不要掛電話！我們把這條 HTTP 連線『升級』成一條專屬的、不斷線的水管 (TCP)。」從此之後，雙方就可以在這條水管裡面用 ReadJSON 和 WriteJSON 瘋狂互丟東西了。

🌟 下一步計畫
